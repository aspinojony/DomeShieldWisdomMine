# 一、项目概述

## 1.1 项目名称

**穹盾智矿（DomeShield Wisdom Mine）**

## 1.2 项目定位

本项目面向**露天矿山 / 露井联采场景**，构建一个集：

- 多源传感器实时监测
- AI 风险识别与时序预测
- 无人机空地协同巡检
- 三维数字孪生可视化
- 生产运营调度
- 后台权限与业务管理

于一体的**综合型智能矿山安全预警与调度平台**。

它不是单纯的“前端大屏”，也不是单一“AI 检测 demo”，而是一套围绕矿山安全与生产管理构建的**端到端系统原型**。

## 1.3 核心价值

本项目重点解决传统矿山管理中以下问题：

1. **监测数据分散**：裂缝、沉降、微震、倾角、气象、巡检图像等数据来源不同，难以统一管理。
2. **异常发现滞后**：人工巡检频次有限，难以及时发现边坡裂缝扩展、局部失稳等风险。
3. **信息难以转化为行动**：很多系统只能“看数据”，不能把异常直接转成任务、工单和调度动作。
4. **演示系统常见问题是“页面会动但数据是死的”**：因此本项目特别强调动态联动与演示数据真实感。
5. **业务系统与可视化系统脱节**：大屏只是展示，后台只是管理，缺少完整的告警→派单→执行→闭环链路。

## 1.4 项目现阶段特点

当前版本已经具备以下鲜明特征：

- 首页大屏支持**动态演示数据联动**，不是纯前端静态 mock。
- 生产运营页已经开始从“图表页面”重构为**调度中枢页面**。
- 已接入 **YOLOv8 裂缝检测模型**，支持无人机视觉链路展示。
- 已构建 **AI 风险预测 + 无人机调度 + 视觉确认 + 前端联动** 的完整演示主线。
- 具备本地开发、Docker 容器部署、演示环境部署三类运行方式。

# 二、建设背景与问题定义

## 2.1 矿山安全场景的复杂性

矿山安全并不是单一指标可以描述的问题，而是一个典型的多因素、多时空尺度耦合问题。典型风险来源包括：

- 边坡裂缝扩展
- 微震能量异常增大
- 地表沉降变化
- 倾角变化 / 结构位移
- 爆破扰动后的应力重分布
- 强降雨 / 大风 / 低能见度导致的外部条件变化
- 运输道路拥堵、作业设备利用率下降等生产侧问题

因此，一个真正有价值的系统必须能够做到：

- 看见异常
- 理解异常
- 判断风险等级
- 决定是否干预
- 触发无人机或人工复核
- 形成闭环记录

## 2.2 传统系统的不足

传统矿山监测平台通常存在如下痛点：

### 1）只有监控，没有决策
很多系统只显示曲线和告警记录，但无法转化为处理任务。

### 2）只有视频，没有分析
监控视频很多，但无法自动识别裂缝、滑坡迹象等异常。

### 3）只有传感器，没有多模态融合
传感器本身会出现误报、噪声或局部偏差，仅靠单源数据难以支撑稳定决策。

### 4）只有 AI 结果，没有业务闭环
AI 算法输出一个“高风险概率”不等于可以直接用于生产管理，必须结合派单、调度、人工确认等环节。

## 2.3 本项目的解决思路

本项目采用“**监测 + 预测 + 复核 + 调度 + 闭环**”的一体化设计：

1. 用传感器持续获取矿区状态；
2. 用时序 AI 模型判断潜在风险趋势；
3. 在高风险情况下触发无人机巡检或人工复核；
4. 用视觉 AI 对图像进行裂缝识别；
5. 将结果映射到业务后台与调度界面；
6. 让异常最终转化为一个可执行、可跟踪、可关闭的事件链路。

# 三、项目目标与设计原则

## 3.1 项目目标

本项目目标不是做一个“功能堆砌型系统”，而是建立一个清晰、可演示、可扩展的矿山智能中枢原型。

### 总目标

构建一套面向矿山安全监测与生产调度的多模块融合系统，实现：

- 数据集中管理
- 风险实时感知
- AI 主动预警
- 无人机协同复核
- 业务调度闭环
- 数字孪生可视展示

### 子目标

1. 建立多服务协同架构；
2. 搭建前端大屏与业务后台；
3. 实现时序风险预测；
4. 实现无人机图像裂缝检测；
5. 建立告警联动与任务闭环；
6. 支持答辩和演示环境快速运行。

## 3.2 设计原则

### 原则 1：可演示优先
系统不仅要逻辑正确，还要能在答辩或展示中稳定呈现核心链路。

### 原则 2：真实感优先
即使部分数据为演示数据，也必须动态变化、符合业务逻辑，不能让人看起来像“死页面”。

### 原则 3：联动优先
不是前端、后端、AI 各做各的，而是强调页面间、服务间、业务间联动。

### 原则 4：分层清晰
UI、业务 API、基础监测 API、AI 引擎、视觉引擎分别承担不同职责。

### 原则 5：便于扩展
允许后续替换数据库、替换模型、接入真实无人机、接入真实 GIS 地图。

# 四、系统总体架构

## 4.1 架构总览

系统采用分层与微服务结合的组织方式，主要由以下几部分组成：

1. **前端可视化与交互层**
2. **业务管理与权限控制层**
3. **监测数据查询与演示数据层**
4. **AI 风险预测层**
5. **视觉识别层**
6. **物联网数据接入与中间件层**
7. **数据存储层**

## 4.2 分层说明

### 1）前端层（frontend_dashboard）
负责：

- 首页大屏展示
- 设备 / 告警 / 生产态势可视化
- 生产运营调度页
- AI 检测与视觉结果页
- 后台登录与权限页面

### 2）业务层（backend_service/business_api.py）
负责：

- 用户认证与权限控制
- 设备台账管理
- 告警规则管理
- 任务 / 工单 / 业务接口
- 生产运营数据聚合输出

### 3）监测数据层（backend_service/api_server.py）
负责：

- 传感器最新值查询
- 历史数据查询
- 演示场景数据构建
- 站点环境模拟接口

### 4）AI 预测层（backend_service/ai_prediction_engine.py）
负责：

- 多模态时序数据建模
- 风险评分预测
- 生成 AI 告警
- 在高风险条件下触发无人机任务

### 5）视觉识别层（ai_core/vision_inference_service.py）
负责：

- 接收图像上传
- 调用 YOLOv8 模型推理
- 输出裂缝识别结果
- 保存分析历史与结果图

### 6）中间件与基础设施层
包括：

- MQTT（EMQX）
- Kafka
- Zookeeper
- InfluxDB
- MySQL
- SQLite

### 7）部署与运维层
包括：

- Docker Compose 编排
- 本地快速启动脚本
- 演示环境 compose 文件
- 脚本化检查与停机工具

## 4.3 系统逻辑链路

一个典型完整链路如下：

1. 传感器数据进入系统；
2. 监测 API 返回最新值；
3. AI 时序引擎读取滑动窗口数据并预测风险；
4. 若风险升高，触发 AI 告警；
5. 高风险时自动调度无人机；
6. 无人机图像进入视觉服务；
7. YOLO 模型输出裂缝检测结果；
8. 前端同步显示告警、任务状态和视觉结果；
9. 业务后台可进行派单、处理、确认闭环。

# 七、核心业务场景

## 7.1 场景一：边坡风险持续监测

系统通过裂缝计、微震仪、倾角计、沉降点等设备持续采样，形成矿区的多模态感知网络。

目的：

- 实时掌握边坡与作业区状态
- 尽早发现结构性变化
- 提供 AI 风险预测输入

## 7.2 场景二：AI 风险升高后的自动响应

当时序 AI 识别到风险概率超过阈值时，系统可以：

- 生成风险告警
- 记录风险证据窗口
- 触发无人机出动
- 联动业务后台生成工单或任务

## 7.3 场景三：无人机图像复核

在高风险区域，系统通过无人机巡检回传图像，并将图片提交至视觉识别服务：

- 检测裂缝数量
- 估算裂缝宽度
- 生成告警级别
- 输出带框佐证图

## 7.4 场景四：生产运营调度

当发现某区域异常后，不仅要“提示风险”，还要：

- 判断是否影响运输 / 作业
- 是否需要暂停某区域工作面
- 是否要派无人机复核
- 是否要生成处置任务
- 是否形成闭环记录

## 7.5 场景五：答辩演示

演示模式下，系统通过动态数据和阶段化场景控制实现：

- 平稳运行
- 先兆变化
- 风险上升
- 无人机调度
- 视觉确认
- 持续观察 / 恢复

这种周期性场景特别适合现场演示与答辩讲解。

# 八、系统功能模块详解

## 8.1 首页总览大屏模块

首页是系统的“态势总控台”，主要展示：

- 设备在线态势
- 告警与预警信息
- 产量与效率
- 任务与运营状态
- 无人机视觉结果
- 三维地图上的风险区域和设备点位

特点：

- 视觉冲击力强
- 动态数据刷新
- 支持地图视角联动
- 支持无人机主线叙事

## 8.2 登录与权限模块

支持基于角色的权限控制。

典型角色包括：

- Admin：管理员
- Engineer：工程师
- Viewer：观察者

权限用途包括：

- 后台页面访问控制
- 告警规则修改权限
- 设备信息增删改查
- 任务状态处理权限

## 8.3 设备台账模块

用于维护矿山设备信息，包括：

- 设备编号
- 设备名称
- 设备类型
- 坐标位置
- 安装位置说明
- 状态信息

作用：

- 为前端地图提供元数据
- 为业务后台提供设备视图
- 为异常定位提供基础数据

## 8.4 告警规则模块

支持通过阈值配置方式建立规则，例如：

- 裂缝宽度 > 10mm
- 沉降值 > 某阈值
- 某类设备状态异常

主要字段：

- `device_id`
- `metric_field`
- `operator`
- `threshold`
- `alert_level`
- `description`

## 8.5 告警记录模块

用于记录告警事件，包括：

- 设备来源
- 指标字段
- 当前值
- 阈值
- 告警等级
- 触发时间
- 是否确认（ACK）

该模块对形成闭环非常重要。

## 8.6 AI 检测与智能识别模块

该模块展示：

- AI 识别结果
- 风险等级
- 视觉历史记录
- 自动巡检最新回传
- 手动上传图像进行裂缝识别

# 十一、AI 引擎详细说明

## 11.1 时序 AI 服务定位

`backend_service/ai_prediction_engine.py` 是项目中的“风险大脑”，主要负责根据多传感器时间窗口判断当前风险等级。

## 11.2 输入特征

当前模型围绕以下 4 类传感器特征组织：

- 裂缝宽度 `crack_width_mm`
- 微震能量 `seismic_energy_j`
- 倾角 `angle_x_deg`
- 沉降值 `settlement_mm`

## 11.3 模型形态

当前代码中使用：

- `MultiSensorLSTM`
- 输入维度 `4`
- 隐层维度 `64`
- 层数 `2`
- 序列长度 `60`

## 11.4 推理机制

### 过程如下：

1. 构建 60 步长数据窗口；
2. 使用 scaler 进行标准化；
3. 转为 Tensor 输入 LSTM；
4. 输出风险概率；
5. 根据阈值转成级别与动作建议。

### 风险阈值逻辑

- `>= 0.85`：警报级（三级）
- `>= 0.50`：预警级（二级）
- `>= 0.20`：监控级（一级）
- 否则为安全

## 11.5 高风险触发逻辑

当风险概率达到高阈值时，系统会自动：

- 记录告警日志
- 触发无人机调度
- 调用业务 API 尝试生成任务 / 工单

这是系统从“预测”走向“行动”的关键环节。

## 11.6 当前项目中的 AI 价值

虽然该模块仍属于原型级实现，但已经具备以下展示价值：

- 能体现多模态融合思想；
- 能展示“从时序数据到风险判断”的逻辑；
- 能作为触发无人机联动的核心引擎；
- 能支撑答辩中关于“主动预警”的叙述。

# 十二、无人机视觉识别链路

## 12.1 模块定位

视觉模块是本项目最直观、最容易体现智能性的部分之一。它负责将无人机获取的现场图像转化为可解释的风险结果。

## 12.2 服务位置

核心服务文件：

- `ai_core/vision_inference_service.py`

模型封装文件：

- `ai_core/models/cv/crack_yolo_model.py`

配置文件：

- `ai_core/configs/cv_yolo.yaml`

## 12.3 输入输出

### 输入

- 无人机侦察图片（HTTP 文件上传）

### 输出

- 检测框列表
- 裂缝数量
- 最大裂缝宽度（估算）
- 告警等级
- 推理耗时
- 结果图路径
- 历史记录条目

## 12.4 推理流程

1. 前端或 AI 服务上传图片到 `/api/v1/vision/analyze_crack`；
2. 服务读取图像并解码；
3. 调用 YOLO 模型推理；
4. 遍历 boxes 生成检测结果；
5. 在原图上绘制框；
6. 将图像保存到 `results/cv/`；
7. 将结构化结果写入 `history.json`；
8. 返回给前端展示。

## 12.5 当前权重接入状态

本项目当前已接入的视觉权重为：

- `ai_core/checkpoints/crack_yolo_best.pt`

该权重已被设置为当前推理优先权重，并且支持通过环境变量：

- `YOLO_WEIGHTS_PATH`

在部署时动态注入。

## 12.6 当前接口

- `GET /api/v1/vision/history`
- `GET /api/v1/vision/latest`
- `POST /api/v1/vision/analyze_crack`
- `GET /health`

## 12.7 当前重要修复点

### 1）前端视觉结果来源修正
之前部分页面错误地把“最新视觉结果”连到了 AI 服务 8001，而不是视觉服务 8003。

现已统一调整为：

- 最新视觉结果由 **8003** 提供；
- 前端按照视觉服务当前返回结构读取。

### 2）权重注入能力增强
`vision_inference_service.py` 已支持通过环境变量注入权重路径，方便部署环境切换模型，而不必改代码。

### 3）推理参数配置化

- 告诉用户下一步该做什么；
- 让系统具备业务操作性；
- 把异常变成任务，把任务变成闭环。

已在项目中形成两份专项设计文档：

### 2）`PRODUCTION-OPS-IMPLEMENTATION.md`
定义了：

- 组件拆分
- 接口映射
- 状态设计
- 开发顺序

## 13.4 目标页面结构

### 顶部：全局态势栏

- 在线设备数
- 执行中任务数

### 左侧：待处理中心
包含：

- 实时告警

### 中间：调度地图
包含：

- 风险区域

根据当前选中的对象显示：

### 底部：处置闭环流水

# 十四、数据流与联动机制

## 14.1 总体数据流

```text
传感器 / 模拟器
   ↓
监测 API / 中间件
   ↓
AI 时序服务
   ↓
告警 / 无人机任务触发
   ↓
视觉服务识别
   ↓
前端首页 / AI 页 / 运营页
   ↓
业务处理 / 派单 / 闭环
```

## 14.2 首页联动机制

首页定时轮询多个服务：

- `8000`：传感器最新值
- `8001`：AI 告警与无人机状态
- `8002`：生产总览与设备元数据
- `8003`：视觉结果

前端再对这些数据进行融合：

- 将设备元数据补充到实时传感器上；
- 根据视觉结果显示无人机联测卡片；
- 根据业务总览刷新 KPI、任务和告警区；
- 根据 AI 状态控制无人机活跃度和提示。

## 14.3 AI → 无人机 → 视觉联动

这是项目最重要的一条链路：

1. AI 风险升高；
2. 生成告警；
3. 调度无人机起飞；
4. 无人机抵近侦察；
5. 图像提交至视觉识别服务；
6. 输出裂缝识别结果；
7. 首页 / AI 页更新视觉结果与风险摘要。

## 14.4 地图联动

当前首页地图支持：

- 主视角
- 无人机视角
- 高危区域视角
- 用户自定义标点与本地保存

这类设计对于答辩非常重要，因为它解决了真实矿区定位难调的问题。

## 17.5 Docker Compose 演示部署方案

文件：`ops/docker-compose.deploy.yml`

特点：

- 更偏向演示环境快速落地
- 只保留核心服务
- 包含：
  - frontend
  - api-server
  - business-api
  - ai-engine
  - vision-engine

### 视觉权重注入

该文件中已为 vision-engine 设置：

```yaml
YOLO_WEIGHTS_PATH: "/app/ai_core/checkpoints/crack_yolo_best.pt"
```

这使得部署时默认加载当前选定的裂缝检测权重。

## 17.6 Docker 构建注意事项

当前项目曾遇到两个实际问题：

### 问题 1：BuildKit 异常
在本机出现过：

- gRPC header 异常
- `non-printable ASCII` 相关错误

解决方式：

- 切换 legacy builder：

```bash
DOCKER_BUILDKIT=0 COMPOSE_DOCKER_CLI_BUILD=0 docker compose -f ops/docker-compose.deploy.yml up -d --build
```

### 问题 2：构建上下文过大
项目中若把大模型、训练压缩包、结果目录都作为上下文，会导致构建缓慢。

当前已新增 `.dockerignore`，用于排除：

- 模型原始素材
- 训练结果
- 大型权重与临时文件
- node_modules / venv / logs 等

# 十九、演示模式与答辩演示建议

## 19.1 为什么要有演示模式

很多课程项目或比赛项目的痛点是：

- 真实设备不一定在现场
- 真实矿区数据无法直接接入
- 网络环境可能不稳定
- 时间有限，无法完整复现真实链路

因此必须提供一个**稳定、可控、看起来真实**的演示模式。

## 19.2 本项目演示模式的特点

### 1）阶段性场景演变
系统会按时间进入：

- 稳定
- 先兆
- 预警
- 调度
- 恢复

### 2）数据是动态变化的
包括：

- 传感器数值
- 产量与效率
- 告警状态
- 无人机状态
- 视觉结果

### 3）能讲清完整故事线
答辩时可按以下顺序讲：

1. 矿区处于稳定运行；
2. 某边坡区域出现裂缝和微震异常先兆；
3. AI 引擎识别到风险升高；
4. 系统自动触发无人机巡检；
5. 无人机图像回传，YOLO 识别到裂缝；
6. 首页与 AI 页面同步显示风险结果；
7. 生产运营页可继续发起任务和闭环处置。

## 19.3 建议的答辩讲解顺序

### 顺序一：先讲问题，再讲系统
- 矿山风险复杂、传统巡检效率低；
- 我们构建了空天地一体化中枢系统。

### 顺序二：展示首页总览
- 介绍多源数据与全局态势；
- 强调不是静态页面。

### 顺序三：展示 AI 联动
- 展示 AI 告警；
- 讲无人机为什么出动。

### 顺序四：展示视觉识别
- 展示裂缝识别结果和证据图；
- 强调 YOLO 已接入真实权重。

### 顺序五：展示运营闭环
- 说明生产运营页如何承接任务和处置。

# 二十一、已知问题与注意事项

## 21.1 前端与接口历史遗留问题

项目在持续迭代中，曾出现以下问题：

1. 部分前端页面路径和接口层级不统一；
2. 某些页面曾错误对接 `8001` 而非 `8003`；
3. 某些旧逻辑假设返回结构是嵌套数据，后续需要持续统一。

## 21.2 Docker 构建问题

本机 Docker BuildKit 存在异常，建议在出现异常时使用 legacy builder。

## 21.3 脚本问题

当前 `scripts/start.sh` 中有一处环境变量写法异常：

```bash
DEMO_MODE=4{DEMO_MODE:-true}
```

这显然应继续修正为标准 shell 写法，否则可能导致启动不符合预期。

## 21.4 数据库与持久化问题

当前很多演示流程更偏向原型系统，正式化时仍需：

- 明确所有业务数据最终落库方案；
- 明确任务、事件、处置流水的持久化模型；
- 明确多用户并发与审计机制。

## 21.5 地图与矿区真实坐标问题

真实矿区定位并不适合长期通过代理盲调，当前已改为支持用户自定义标点，这种方案更适合继续保留。
