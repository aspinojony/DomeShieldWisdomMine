# 穹盾智矿（DomeShield Wisdom Mine）项目文档

## 一、项目简介

穹盾智矿是一套围绕 **矿山安全预警 + 无人机视觉复核 + 生产运营调度 + 三维态势展示** 构建的综合演示系统。项目目标不是只做一个“会动的大屏”，而是形成一条完整业务链路：

**监测数据采集 → 风险研判 → AI / 无人机复核 → 调度与处置 → 前端联动展示**

系统同时覆盖了两个层面：

1. **展示层**：用于大屏演示、项目汇报、竞赛答辩
2. **业务层**：用于设备、告警、权限、任务等业务管理

---

## 二、建设目标

本项目重点解决以下几个问题：

### 1. 矿山风险监测数据分散
裂缝、微震、倾角、沉降、视频巡检等信息通常分散在不同系统中，缺乏统一展示和统一判断。

### 2. 告警缺少联动闭环
很多系统只能“报异常”，但不能完成：
- 定位异常区域
- 判断严重程度
- 派发任务
- 跟踪执行
- 形成闭环

### 3. 演示场景普遍假数据感强
传统竞赛型系统常见问题是页面很炫，但数据是死的。穹盾智矿在本轮重构中重点加强了**动态演示可信度**：
- 演示数据随时间波动
- 告警会阶段性出现
- 无人机会联动状态变化
- 视觉结果可以真实由识别服务返回

                    │ HTTP / REST

                    │

## 5.2 后端模块

## （1）监测接口服务 `backend_service/api_server.py`（8000）
主要职责：
- 提供传感器最新值接口
- 提供历史数据接口
- 在依赖缺失时返回演示数据
- 为首页与图表提供实时数据支撑

关键接口：
- `GET /health`
- `GET /sensors/latest`
- `GET /api/v1/sensors/latest`
- `GET /api/v1/sensors/history/{device_id}`

本轮改动重点：
- 增加 `/sensors/latest` 别名，兼容前端旧写法
- 演示数据改为随时间变化，不再固定死值
- 风险阶段采用周期场景机制驱动

## （2）业务平台服务 `backend_service/business_api.py`（8002）
主要职责：
- 登录鉴权
- 用户与角色管理
- 设备台账管理
- 告警规则与告警记录管理
- 生产运营总览数据提供

关键能力：
- 默认管理员自动初始化
- JWT 登录鉴权
- 基于角色的接口保护
- 生产运营摘要接口用于首页和运营页联动

典型接口：
- `POST /api/v1/auth/login`
- `GET /api/v1/devices`
- `GET /api/v1/alert-rules`
- `GET /api/v1/ops/mining-summary`

## （3）AI 风险引擎 `backend_service/ai_prediction_engine.py`（8001）
主要职责：
- 模拟/执行时序风险推演
- 根据风险阶段产生告警
- 维护无人机状态变化
- 在高风险时触发无人机复核链路

关键能力：
- 风险周期模拟
- 多阶段状态推进
- 无人机起飞 / 抵近 / 返航状态流转
- 可触发业务服务创建任务
- 可调用 8003 视觉分析接口

典型接口：
- `GET /api/v1/alerts/latest`
- `GET /api/v1/drones/status`
- `GET /api/v1/vision/latest`（AI 侧兜底视觉摘要）
- `POST /api/v1/ai/trigger_crisis`

## （4）视觉识别服务 `ai_core/vision_inference_service.py`（8003）
主要职责：
- 接收图片
- 调用 YOLOv8 模型推理
- 输出识别结果与落盘结果图
- 维护历史识别记录
- 为前端提供历史 / 最新结果查询

关键接口：
- `GET /health`
- `GET /api/v1/vision/history`
- `GET /api/v1/vision/latest`
- `POST /api/v1/vision/analyze_crack`

返回信息包括：
- 识别时间
- 推理耗时
- 裂缝数量
- 最大裂缝宽度
- 告警等级
- 结果图 URL
- 检测框详情

## 六、模型与算法说明

## 6.1 时序风险模型
项目中包含时序模型相关代码，用于融合多源传感器指标并给出风险概率判断。当前演示环境下，更多承担的是**风险场景驱动与联动演示**角色。

目标字段通常包括：
- 裂缝宽度
- 微震能量
- 倾角变化
- 沉降量

输出结果包括：
- 风险概率
- 风险等级
- 建议动作

## 6.2 YOLO 裂缝检测模型
当前视觉服务使用 YOLOv8 做裂缝识别，支持：
- 自定义权重加载
- 结果图输出
- 历史记录保存
- 告警等级换算

当前实际接入权重：
- `ai_core/checkpoints/crack_yolo_best.pt`

当前配置文件：
- `ai_core/configs/cv_yolo.yaml`

支持通过环境变量覆盖：
```bash
YOLO_WEIGHTS_PATH=/app/ai_core/checkpoints/crack_yolo_best.pt
```

## 八、典型业务流程

## 8.1 首页演示流程
1. 首页加载后读取 8002 的矿山总览数据
2. 读取 8000 的传感器最新值
3. 读取 8001 的 AI 风险与无人机状态
4. 读取 8003 的最新视觉结果
5. 前端将这些数据融合后展示为动态总览

## 8.2 无人机视觉复核流程
1. AI 风险引擎识别到高风险
2. 无人机状态切换为调度/侦察状态
3. 视觉服务接收图片并完成识别
4. 输出结果图和识别指标
5. 前端读取最新视觉结果并展示联测卡片

## 8.3 人工上传识别流程
1. 用户在智能识别页上传图片
2. 前端调用 `POST /api/v1/vision/analyze_crack`
3. 视觉服务返回识别结果
4. 保存到历史记录
5. 前端刷新历史结果列表

## 九、部署方式

## 9.1 本地开发部署
适合：
- 联调
- 页面开发
- 视觉模型调试
- 课堂/实验室环境演示

前端：
```bash
cd frontend_dashboard
npm install
npm run dev
```

后端 / AI：
```bash
uvicorn backend_service.business_api:app --host 0.0.0.0 --port 8002
python backend_service/api_server.py
python backend_service/ai_prediction_engine.py
python ai_core/vision_inference_service.py
```

## 9.2 Docker 演示部署
适合：
- 本机一键拉起
- 服务器快速部署
- 答辩现场部署

```bash
cd /path/to/project
DOCKER_BUILDKIT=0 COMPOSE_DOCKER_CLI_BUILD=0 docker compose -f ops/docker-compose.deploy.yml up -d --build
```

说明：
- 关闭 BuildKit 是为了规避当前环境下偶发的 gRPC header 异常
- 首次构建会较慢，主要耗时在 PyTorch / OpenCV 依赖安装

## 9.3 基础设施部署
如果需要更完整的消息与数据库能力，可使用：
- `ops/docker-compose.yml`

它包含：
- EMQX
- InfluxDB
- MySQL
- Zookeeper
- Kafka
- 前端
- 后端
- AI 服务

## 十二、当前存在的问题与注意事项

### 1. 项目仍偏演示原型，不是完整工业生产系统
目前更适合：
- 答辩展示
- 原型验证
- 功能联调
- 方案汇报

### 2. 部分本地脚本需要根据环境调整
尤其是：
- Python 虚拟环境路径
- Node 依赖路径
- Docker 构建方式

### 3. 大模型/训练素材不适合全部入 Git
`模型/` 目录体积大，应与推理权重分开管理。

### 4. 首页虽然已动态化，但仍建议后续继续做真实业务接口统一
现在部分场景仍采用演示驱动逻辑，后续如要工程化，需要进一步统一接口来源与状态机。
